
# 统一信用点数支付网关 - 产品需求文档 (PRD) 
**版本:** v1.0  
**状态:** 草案

## 1. 产品定位
本产品是一个对标 Stripe/PayPal 体验的**统一信用点数（Credit）支付网关**。
旨在为商户（包括人类开发者和 AI 智能体）提供极简的 API 与数字服务变现工具，同时为消费者提供“无感点数扣费”与“访客法币直付”双轨并行的丝滑消费体验。底层架构逻辑对齐 x402 标准，未来可平滑升级兼容 Web3 与区块链结算。

## 2. 核心特征 (Core Characteristics)

* **双轨收银台：** C 端用户在支付时，既可以登录账号使用 Credit 余额秒级支付，也可以作为纯访客直接刷信用卡/微信/支付宝完成支付。系统将在后台自动为商户折算为 Credit。
* **统一资产单位：** B 端商户的账户看板永远只接收和显示“Credit”，彻底屏蔽复杂的法币汇率、手续费计算和支付渠道差异。
* **极简接入标准：** 万物皆基于“拒绝访问并明码标价（类 HTTP 402 逻辑）”，商户仅需设定资源价格并接收支付成功通知，无需关心用户的实际支付路径。
* **微支付友好：** 依靠内部中心化高并发账本实现真正的“按次/按需计费”，打破传统法币系统高昂手续费导致的商业模式限制（针对内部 Credit 闭环生态）。

## 3. 关键功能场景 (Functional Scenarios)

### 3.1 消费者场景 (C端)

**场景一：访客极速支付 (Guest Checkout)**
* **触发机制：** 用户在商户 A 的网站点击“生成行业报告”（标价 100 Credits）。
* **用户体验：** 用户跳转至官方收银台。页面提示“商户 A 请求收取 \$1.00”。用户无需注册登录，直接在页面输入信用卡号或使用快捷支付完成付款。
* **预期结果：** 支付成功，用户被重定向回商户 A 网站并立刻获取报告。

**场景二：注册用户余额支付 (Logged-in Checkout)**
* **触发机制：** 用户在商户 B 的网站点击“购买单次 API 调用”（标价 50 Credits）。
* **用户体验：** 跳转至收银台后，系统识别到用户已登录，页面直接显示“当前余额 500 Credits，本次扣除 50 Credits”。用户点击“确认”。
* **预期结果：** 余额秒级扣除，瞬间跳回商户 B 网站获得服务。

**场景三：AI 智能体/开发者自动代扣 (API Key 支付)**
* **触发机制：** 用户的 AI 助手在后台高频调用各类商户的计费 API。
* **用户体验：** 用户在平台控制台生成一把“全局 API Key”并配置给 AI。AI 每次发请求时携带此 Key，网关在后台自动扣除对应 Credit，余额不足时自动阻断请求。

**场景四：用户充值中心 (Top-up)**
* **触发机制：** 用户主动补充账户余额。
* **用户体验：** 用户登录平台，使用法币购买 Credit 充值包（如：支付 \$10 获得 1000 Credits），并在个人中心查看跨商户的详尽消费流水。

---

### 3.2 商户场景 (B端 / AI Agent)

**场景一：商户入驻与服务定价**
* **触发机制：** 开发者或 AI 程序需要变现其服务。
* **用户体验：** 商户在平台注册，获取专属 `商户 ID`。商户在其服务端代码中设定规则：“若请求无支付凭证，则返回包含价格（如 50 Credits）的报错响应”。

**场景二：发起收款与履约**
* **触发机制：** 响应消费者的购买需求。
* **用户体验：** 商户将用户引导至平台的收银台链接。用户完成付款后，商户的服务器（或 AI 的后台程序）将立刻收到一条标准化的 Webhook“支付成功”异步通知。商户验证通知后，自动放行资源或服务。

**场景三：资产管理与防欺诈**
* **触发机制：** 商户查看营收与对账。
* **用户体验：** 无论消费者使用何种法币渠道支付，商户后台的看板仅显示“收入增加 50 Credits”。商户无需处理信用卡拒付（Chargeback）、无需承担跨国汇率损失及坏账风险。

**场景四：微支付防亏损策略设置**
* **触发机制：** 拦截高成本的低客单价法币支付。
* **用户体验：** 商户可在后台配置“访客法币支付最低门槛”（例如：低于 500 Credits 的订单隐藏法币直付选项，强制要求访客注册并充值），以避免法币渠道固定手续费造成的“倒贴”亏损。
